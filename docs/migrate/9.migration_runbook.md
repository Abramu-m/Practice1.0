# Online Migration Runbook

End-to-end order of operations to stand up the new online Practice1.0/Janet database,
combining setup data carried from local with transactional data pulled from Medcom-online.
See `7.online_migration_categorization.md` for the table categorization this is based on.

## 1. Provision the new online database (empty schema)

On the online server, create the database and run migrations to build an empty schema
(this already includes the `store_stock_movements` FK fix from migration
`2026_06_12_120000_drop_batch_id_from_store_stock_movements_table`, baked into
`database/schema/mysql-schema.sql`):

```
php artisan migrate --force
```

## 2. Import setup/configuration data (Section A)

On the local XAMPP machine, export Section A tables:

```
bash docs/migrate/8.setup_export.sh > 8.setup_data.sql
```

Copy `8.setup_data.sql` to the online server and import it into the new (empty) schema:

```
mysql -u<user> -p <online_db> < 8.setup_data.sql
```

This populates staff accounts, facility config, pricing/NHIF/insurance maps, CDS rules,
catalogs (medications, services, ICD-10/MTUHA/IDSR/MSD/LOINC), and report templates —
all with their original local IDs preserved (required for the insurance map FKs).

## 3. Run the transactional migration scripts (Section B)

Medcom-online and the new `practice` database are on the same MySQL server, so the
adapted scripts in `docs/migrate/online/` run as-is via cross-database
`INSERT ... SELECT` — source stays `yyfcolmy_medcom`, target is `yyfcolmy_brigita_practice`.

These are NOT the original `1.first.sql`/`2.results.sql`/`3.third.sql` — every INSERT
whose target table is in Section A (already populated by step 2) has been removed,
including the 9 statements that originally sourced from a third schema
(`yyfcolmy_practice1_0`, an earlier online deployment). See note 3 and 5 in
`7.online_migration_categorization.md` for details, and re-run the spot-check in
note 5 before this step.

Run, **in this order**, against the new online `yyfcolmy_brigita_practice` schema:

1. `docs/migrate/online/1.first.sql` — patients, patient_visits, consultations,
   financial_transactions, goods_received_notes/items, icd_diagnoses,
   past_medical_history, investigations
2. `docs/migrate/online/2.results.sql` — investigation_template_results (large)
3. `docs/migrate/online/3.third.sql` — prescriptions, medication_ledger,
   vital_signs, systemic_examinations, store_locations_stock

`4.pricing.sql`, `5.nhif_mapping.sql`, `6.mtuha_mapping.sql` are **not** re-run here —
their output (tariff tables, insurance maps, ICD-10 MTUHA mapping) was already carried
over as setup data in step 2.

## 4. Post-import backfill (required)

Raw-SQL inserts in step 3 don't fire model observers, so run (per
`docs/todo_after_migrate.md`):

```
php artisan medications:backfill-legacy-cash-sales --dry-run
php artisan medications:backfill-legacy-cash-sales

php artisan finance:backfill-legacy-transactions --dry-run
php artisan finance:backfill-legacy-transactions

php artisan investigations:backfill-tb-leprosy-results --dry-run
php artisan investigations:backfill-tb-leprosy-results

php artisan investigations:backfill-procedure-images --dry-run
php artisan investigations:backfill-procedure-images

php artisan investigations:backfill-cd4-results --dry-run
php artisan investigations:backfill-cd4-results

php artisan investigations:backfill-bedrest-observation --dry-run
php artisan investigations:backfill-bedrest-observation
```

The `backfill-tb-leprosy-results` command re-extracts
`{legacy_medcom_database}.tb_leprosy_form` (TB/Leprosy microscopy, Xpert MTB/RIF and
skin-smear results) into `investigation_template_results` using the `genxpert_tb`
template structure, overwriting the generic `'One Line'` rows that `2.results.sql`
created for the same investigations. It also populates `investigation_form_data` with
the original request-side fields (specimen type, reason for examination, HIV status,
test(s) requested, etc.) so the REQUEST section of the `genxpert_tb` form renders with
the original request data.

The `backfill-procedure-images` command copies legacy Procedure/Specialized
Investigations result images (`{legacy_medcom_database}.procedures.pimage`, e.g. X-Ray
films and ECHO stills) from the legacy Medcom1.0 app's `production/uploads` directory
into `storage/app/public/investigation_results`, and links the new paths into
`investigation_template_results.form_data['images']` for the matching investigation.
It requires filesystem access to the legacy uploads directory — set
`LEGACY_PROCEDURE_UPLOADS_PATH` if it isn't at the default sibling
`../Medcom1.0/production/uploads` path on that server. Most `pimage` values are
placeholders with no real file and are skipped automatically; the command is
idempotent and safe to re-run.

The `backfill-cd4-results` command re-extracts `{legacy_medcom_database}.cd4_form`
(CD4 request indication, CTC number, and result) into `investigation_template_results`
using the `cd4` template structure, overwriting the generic `'Single Numeric Lab
Values'` rows that `2.results.sql` created for finalized CD4 investigations. Rows with
no result yet (`formstatus` 1/2) are also created as `draft`/`preliminary` so the CD4
indication and CTC number are pre-filled when the result is entered.

The `backfill-bedrest-observation` command migrates `{legacy_medcom_database}.bedrest_observation`
(vitals recorded during "Bed Rest/Observation" services) into `investigation_template_results`
using the `vital_observations` template structure (the `medical_services` row for "Bed
Rest/Observation" is reassigned from `narrative_lab` to `vital_observations` by migration
`2026_06_13_064604_reassign_bedrest_observation_result_template`). For investigations with
multiple legacy readings, the latest reading is used for the vitals snapshot and all readings'
timestamped comments are merged into `nursing_notes`. All migrated rows are marked `final`.

## 5. Leave empty (Sections C & D)

No action needed — NHIF claims, dispensing/sales, store requisitions/stock
movements/corrections, payment receipts, allergies, referrals, CDS alerts, sync
tables, NHIF member cache, and all framework infrastructure tables start empty and
populate naturally as the new system is used. (`goods_received_notes`/
`goods_received_note_items` are populated in step 3, not here.)

## 6. Verify

- Spot-check FK integrity: `medication_insurance_map`/`medical_service_insurance_map`
  rows resolve to real `medications`/`medical_services` IDs.
- Confirm `medications.stock_quantity` matches the sum of `store_locations_stock`
  for that medication (both should reflect the Medcom-sourced figures from step 3).
- Log in with a carried-over `users` account and confirm role/permissions look right.
- Spot-check a recently created patient/visit in Medcom-online appears in the new DB.
- Compare row counts between `yyfcolmy_medcom.pat_visit`/`pat_lab`/`pat_pharm` and the
  new `patient_visits`/`investigations`/`prescriptions` — a shortfall indicates rows
  were dropped by the `INNER JOIN`s in step 3 because of the ID-alignment gap noted
  in `7.online_migration_categorization.md` note 5 (e.g. a doctor, medication, or
  service added in Medcom after the local DB was last synced).
- Confirm `goods_received_notes`/`goods_received_note_items` populated and that
  `goods_received_note_items.item_id` resolves to real `medications.id` values.
