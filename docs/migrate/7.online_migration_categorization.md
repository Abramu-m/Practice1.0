# Online Migration: Setup (local) vs Transactional (Medcom online)

Goal: stand up the new online Practice1.0/Janet database using two sources:

- **Setup/configuration** — carried as-is from the local XAMPP `practice` database (pricing, NHIF/MTUHA mappings, CDS rules, catalogs, staff accounts, facility config).
- **Transactional/day-to-day data** — pulled fresh from the **online legacy Medcom** database, using the same approach as `1.first.sql` / `2.results.sql` / `3.third.sql`, but with Medcom-online as the source instead of local `medcom1_0`.

All 109 application tables in `database/schema/mysql-schema.sql` are accounted for below.

---

## A) SETUP/CONFIG — carry as-is from local `practice` DB

Export with `mysqldump --no-create-info` (target schema already exists via migrations), **preserving IDs** — several of these are FK-referenced by other tables in this section (e.g. insurance maps reference `medications.id` / `medical_services.id`).

**Users / roles / facility**
- `users`, `doctors`, `designations`, `facilities`, `nhif_settings`

**Patient/visit configuration**
- `patient_categories`, `visit_types`, `age_groups`, `patient_category_visit_type`, `consultation_fees`

**Medication catalog** — `medications` carried with **all columns except** `stock_quantity`, `reorder_level`, `maximum_stock_level` (set to 0/defaults on import; real values come from Section B via `medication_ledger`/`store_locations_stock`)
- `medications` (partial), `medication_formulations`, `medication_units`, `medication_frequencies`, `administration_routes`, `drug_classes`, `drug_class_medication`

**Medical services / lab catalog**
- `medical_services`, `service_categories`, `sample_types`, `result_templates`, `investigation_forms`, `investigation_consumables`

**Clinical coding**
- `icd_10` (incl. `mtuha_diagnosis` mapping column already populated locally), `mtuha_diagnoses`, `idsr_diagnoses`, `idsr_icd_mapping`, `msd_codes`, `lab_codes`

**Store/inventory definitions (not stock levels)**
- `store_locations`, `store_units`, `store_suppliers`, `store_categories`

**NHIF/insurance configuration**
- `nhif_tariffs`, `medication_insurance_map`, `medical_service_insurance_map`
- Per-insurance tariff tables created by `4.pricing.sql`: `assemble_tariff`, `jubilee_tariff`, `shib_tariff`, `strategies_tariff`

**Clinical Decision Support (CDS) configuration**
- `cds_rule_categories`, `cds_rule_types`, `cds_rules`, `cds_rule_conditions`, `cds_rule_parameters`, `cds_dosage_limits`

**Referral configuration**
- `referral_hospitals`, `referral_departments`

**Report templates & system settings**
- `system_settings`, `blood_transfusion_report_rows`, `hematology_report_rows`, `clinical_chemistry_report_rows`, `serology_report_rows`, `microbiology_report_rows`, `parasitology_report_rows`, `medicine_dispensing_report_rows`

---

## B) TRANSACTIONAL — source from Medcom-online

Re-run the logic of `1.first.sql`, `2.results.sql`, `3.third.sql` with the source schema pointed at the **online Medcom** database and the target at the new online `practice` database.

> If Medcom-online and the new Practice1.0 DB are **not** on the same MySQL server, the source tables must first be dumped/imported into a staging schema on the target server, since these scripts rely on cross-database `INSERT ... SELECT`.

- `patients`, `patient_visits`, `vital_signs`
- `consultations`, `icd_diagnoses`, `systemic_examinations`, `past_medical_history`
- `investigations`, `investigation_consumptions`, `investigation_form_data`, `investigation_template_results`
- `prescriptions`
- `medication_ledger`, `store_locations_stock` — **this is the source of current stock**, confirming the exclusion of stock columns in Section A
- `financial_transactions`
- `goods_received_notes`, `goods_received_note_items` — sourced from `grn_hd`/`grn_dt` in `1.first.sql`; moved here from Section C (see note 4 below)

**Required post-import step** (per `docs/todo_after_migrate.md`), run in order:

```
php artisan medications:backfill-legacy-cash-sales --dry-run   # then for real
php artisan finance:backfill-legacy-transactions --dry-run     # then for real
```

Needed because raw-SQL inserts don't fire model observers, so `financial_transactions` and OTC cash sales rows won't otherwise exist.

## C) START EMPTY — Practice1.0-only, no Medcom equivalent

These tables have no source in legacy Medcom and start empty in the new online DB (they populate naturally going forward):

- **NHIF claims**: `nhif_claims`, `nhif_claim_items`, `nhif_claim_diseases`, `nhif_claim_batches`, `nhif_claim_feedback`, `nhif_claim_errors`
- **Dispensing/sales**: `medication_cash_sales`, `medication_cash_sale_items`
- **Inventory operations**: `store_requisitions`, `store_requisition_items`, `store_stock_movements`, `stock_corrections`, `unfit_medications`, `reconciliation_runs`
- **Finance**: `payment_receipts`, `daily_cash_reconciliation`
- **Clinical**: `allergies`, `patient_referrals`, `procedure_consumptions`, `cds_alerts`, `cds_alert_actions`
- **Offline-first sync**: `sync_outbox`, `sync_state`, `sync_conflicts`

## Notes / things to double-check before running the real export

1. **Preserve IDs in Section A.** `medication_insurance_map` / `medical_service_insurance_map` reference `medications.id` / `medical_services.id` from the *same* local DB — if AUTO_INCREMENT is reset or IDs are remapped during import, these maps will point at the wrong rows.
2. **Schema bug found while reviewing — fixed.** `store_stock_movements` had a foreign key `store_stock_movements_batch_id_foreign` referencing `store_stock_batches(id)`, but `store_stock_batches` did not exist anywhere in `database/schema/mysql-schema.sql`. Fixed via migration `2026_06_12_120000_drop_batch_id_from_store_stock_movements_table` (dropped the FK and the `batch_id` column); `mysql-schema.sql` has been regenerated.
3. **Third schema discovered: `yyfcolmy_practice1_0`.** `1.first.sql`/`3.third.sql` source several setup tables — `visit_types`, `cds_rule_categories`, `cds_rule_types`, `drug_classes`, `store_units` (1.first.sql) and `mtuha_diagnoses`, `medication_formulations`, `result_templates`, `nhif_tariffs` (3.third.sql) — from an apparent prior online Practice1.0 deployment, not from Medcom or local. These are all Section A tables already covered by the local export, so the adapted scripts in `docs/migrate/online/` drop these INSERTs entirely.
4. **`goods_received_notes`/`goods_received_note_items` reclassified B (not C).** The original Section C placement assumed no Medcom source, but `1.first.sql` populates both from `grn_hd`/`grn_dt`. Kept in the adapted `online/1.first.sql`.
5. **ID-alignment assumption for the adapted Section B scripts.** `docs/migrate/online/1.first.sql` and `online/3.third.sql` drop every INSERT whose target table is in Section A (whether originally sourced from `yyfcolmy_medcom` or `yyfcolmy_practice1_0`), since those tables are now populated by the local export (`8.setup_export.sh`). The remaining Section B inserts reference those tables via raw legacy IDs or joins (e.g. `doctors.doctor_id`, `medications.id`, `medical_services.id`, `store_locations.id`, `store_suppliers.id`, `icd_10.code`, `administration_routes.id`, `medication_frequencies.id`). This only works if the local copies of these tables still have the **same IDs/codes** as the current `yyfcolmy_medcom` reference tables. Verified for `doctors`, `medications`, `administration_routes` (local IDs have gaps consistent with preserved medcom IDs). **Before running `online/1.first.sql`/`online/3.third.sql`, spot-check whether `yyfcolmy_medcom.doctor`/`drugs`/`services`/`section`/`supplier`/`frequency`/`icd_mst` have any rows added since the local `practice` DB was last synced — if so, those new rows (and any transactional records referencing them) will be silently dropped by the `INNER JOIN`s.**
